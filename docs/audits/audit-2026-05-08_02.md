# Codebase Audit Report: Lunaris Play Architecture

**Date:** May 8, 2026  
**Auditor:** Cypher (Refactoring Strategist)  
**Status:** 🚨 **Immediate Refactoring Recommended**

---

## 1. Executive Summary

The current architecture of Lunaris Play is a **Modular Monolith with High Coupling**. While the logic is functional for two games (Piou Piou and Dixit), the codebase exhibits significant "Gravity" toward a **God Object** pattern in the `rooms` table. Adding a third game in the current state will result in a combinatorial explosion of `if/else` statements in the core engine, making a UI/UX overhaul extremely risky and brittle.

---

## 2. Full-Stack Data Flow Mapping

1.  **Schema Layer (`schema.ts`)**: Defines the source of truth. The `rooms` table acts as a global state bucket.
2.  **Orchestration Layer (`engine.ts`)**: Handles room lifecycle (Join, Start). It is currently polluted with game-specific initialization logic.
3.  **Domain Logic Layer (`games/*.ts`)**: Contains the rules. However, there is no formal interface, leading to inconsistent implementations of turn management and scoring.
4.  **Hook Layer (`useGame.ts`)**: Consumes `getRoomState`. It performs manual derivation of state (e.g., `isMyTurn`), which should ideally be pre-calculated or strictly typed.
5.  **UI Layer**: (Inferred) Likely heavy on conditional rendering based on `room.currentGame` slug, making the frontend a reflection of the backend's coupling.

---

## 3. Technical Debt & Anti-patterns

### 🚨 Anti-pattern: The "Switch-Case" Engine
**Files:** `convex/engine.ts`, `convex/game.ts`  
**Description:** The core engine explicitly checks for `gameSlug === "dixit"` or `"pioupiou"`.  
**Impact:** Violation of the **Open/Closed Principle**. Every new game requires invasive changes to core infrastructure files.

### 🚨 Anti-pattern: The Polymorphic Blob (God Object)
**Files:** `convex/schema.ts` (`rooms.gameBoard`, `players.state`)  
**Description:** Instead of game-specific tables or a clean union type, the schema uses optional fields and `v.any()`.  
**Impact:** Total loss of type safety. A bug in "Dixit" logic could theoretically corrupt "Piou Piou" state because they share the same object structure.

### 🚨 Anti-pattern: Fragmented State Transitions
**Files:** `convex/games/pioupiou.ts` (`finishTurn`), `convex/games/dixit.ts` (`calculateScores`)  
**Description:** Each game reimplements the "End of Turn" logic (patching `currentTurnIndex`, `history`, `status`).  
**Impact:** High risk of "Zombie Rooms" where a game finishes but the status isn't updated correctly, or turn order skips players due to slight logic variations.

---

## 4. Structural Weaknesses (UI/UX Overhaul Blockers)

1.  **Opaque Game State**: The frontend receives a `gameBoard` object but doesn't know its "Shape" without checking the slug. This forces the UI to be a series of `if (dixit) <DixitUI /> else ...`.
2.  **Lack of Action Validation**: There is no centralized way to validate if an action is legal for the current game phase *before* executing it.
3.  **Type-Casting Debt**: The frontend hook `useGame.ts` uses `(api as any)` and `p: any`. This masks breaking changes during refactoring.

---

## 5. Strategic Refactoring Roadmap

### Phase 1: Standardized Game Interface (Immediate)
Define a `GamePlugin` interface on the backend.
- `init(players: ID[]): BoardState`
- `onJoin(player: ID): PlayerState`
- `handleAction(ctx, state, action): NewState`
- Move all logic out of `engine.ts` into these plugins.

### Phase 2: Schema Hardening
- Transition `gameBoard` and `playerState` to use a `v.union` of strict objects.
- Ensure Convex types are generated and used without `any` in the frontend.

### Phase 3: Centralized State Machine
- Implement a core `Transition` helper that handles:
    - Turn advancement
    - History logging
    - Win condition checking
- This ensures consistency across all games.

---

## 6. ADR-001: Plugin-Based Game Architecture

### Status
**Proposed**

### Context
The `engine.ts` and `game.ts` files are tightly coupled to specific game implementations, making the system difficult to scale and test.

### Decision
We will move toward a "Plugin" architecture where `engine.ts` only handles generic room logic (CRUD) and delegates game-specific behavior to a registry of game modules.

### Consequences
- **Easier**: Adding a new game (e.g., "7 Wonders") only requires adding a new file in `convex/games/`.
- **Harder**: Requires a one-time effort to abstract the "Turn" and "Action" concepts into a generic format.

---

**Cypher's Verdict:** The foundation is currently "Spaghetti-on-Convex". Before proceeding with a massive UI/UX overhaul, the backend must be decoupled, or you will find yourself fighting the data model at every step of the design.
