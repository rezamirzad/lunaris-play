# Codebase Audit Report: PiouPiou Game Logic and Room Routing

**Date:** May 8, 2026
**Auditor:** Cypher

## đź“Š System Status Report & Technical Debt Identification

### 1. `frontend/src/app/room/[roomId]/page.tsx`
* **Architectural Flaw:** Monolithic routing component. The component relies heavily on conditional rendering based on query parameters (`?view=board`) and the `room.currentGame` state to determine which game's components to load.
* **Code Smell (OCP Violation):** Hardcoded conditional rendering for games (`gameType === "dixit" ? ... : ...`). This tightly couples the generic room page to specific game implementations. Every new game added requires modifying this file, preventing scalable expansion.
* **Code Smell:** Deriving player identity via `localStorage.getItem("playerName")` inside a `useEffect`. This delays identity resolution, forcing an "Authenticating..." flash on every load, and is brittle against user tampering. 

### 2. `convex/games/pioupiou.ts`
* **Architectural Flaw:** The `handleAction` mutation is a "God Function." It attempts to handle every possible game action (Lay, Hatch, Discard, Defend, Accept, Attack) inside a single massive control flow block.
* **Code Smell:** Weak typing on the mutation arguments. `actionType` is typed as an optional string (`v.optional(v.string())`), and the validation logic is based on manually checking lengths and contents of the `cards` array. This bypasses Convex's strong schema validation capabilities in favor of complex, error-prone branching logic.
* **Maintenance Bottleneck:** Business rules (game mechanics) are deeply intertwined with data access (Convex `ctx.db` calls). This makes unit testing the game logic without a database difficult and complicates future rule adjustments.

## đź—şď¸Ź Strategic Improvement Roadmap (UI/UX Readiness)

The current architecture is **poorly prepared** for a massive UI/UX overhaul. If the UI team wants to add complex interactions (e.g., dragging a Fox to attack a specific player with optimistic UI updates), the frontend will struggle. It currently relies on passing opaque arrays of indices and strings to a monolithic backend function that lacks distinct, strongly-typed action boundaries.

**Phase 1: Decouple Game Implementations (Frontend)**
* Create a generic `GameRenderer` component or registry. Game components should register themselves dynamically (e.g., mapping game IDs to components) rather than hardcoding conditionals in the top-level page.
* Consider moving the board/player view dichotomy into nested routes (e.g., `/room/[roomId]/board` and `/room/[roomId]/player`) or use Next.js parallel routes to decouple their lifecycles.

**Phase 2: Action Segregation (Backend)**
* Split `handleAction` into smaller, distinct, well-typed mutations (e.g., `hatchEgg`, `playFox`, `blockFox`). 
* Use Convex's `v.union` and discriminated unions for strongly typed payload validation on the backend APIs.

**Phase 3: Domain Extraction**
* Extract pure game logic (e.g., calculating validity of a move or next state) out of the mutations into pure TypeScript functions. These functions should take the current game state and return the next state, leaving the Convex mutations to act purely as orchestrators for DB fetching and patching.

## đź“‹ Architecture Decision Record

# ADR-001: Extract Polymorphic Game Actions to Distinct Mutations

## Status

Proposed

## Context

The `pioupiou.ts` game engine currently relies on a single `handleAction` mutation that evaluates all game state transitions via complex `if/else` logic and weakly typed generic payloads. This creates a monolithic chokepoint that is hard to maintain, test, and adapt for granular UI interactions. The frontend is forced to bundle different actions into generic argument shapes.

## Decision

We will deprecate the single `handleAction` mutation and replace it with a suite of cohesive, strongly typed mutations specific to each explicit game action (e.g., `layEgg`, `hatchEgg`, `attackWithFox`, `defendAttack`). Pure game logic will be separated from data access into testable helper functions.

## Consequences

* **Easier:** Validating specific actions, debugging edge cases, creating granular UI feedback (optimistic updates), and unit testing pure state transitions.
* **Harder:** Initial refactoring will require updating all frontend call sites to use the new specific mutations instead of the generic payload dispatcher, and ensuring atomic updates across potentially overlapping actions.
